# 真人对话「回复教练」：听对方 → 给多条可选答句（中/日/读音）

- **日期**：2026-07-16
- **状态**：待验证
- **标签**：product, language-learning, speech, AI, AdventureX
- **提出人**：千坂神奈子（与路路讨论）

## 背景

刷到日本留学生找便利店兼职、日语不好的视频；字幕里的日语对话让人联想到同声传译。现有 AirPods 等即时翻译能「听懂」，但留学场景更需要**自己开口说**——听懂了却不知道怎么答，仍是障碍。

想做的不是纯翻译，而是：**听对方 → AI 给出数条可说的答句 → 用户自己选一句念出去练语言**。

## 想法（她真正想做什么）

### 一句话

面向真实对话（便利店、电话、会议）的**语言开口教练**：识别对方说的话，生成多条回复候选（母语释义 + 目标语 + 读音标注），用户挑选后自己念；可选识别用户念了哪句，并记住对话流以改进后续建议。

### 核心流程（核心功能）

1. **听对方**：语音识别对方内容（不是听自己）
2. **出题**：根据对方话 + 对话记忆，生成 **N 条**回复候选
3. **三语/三层展示**：中文意思、日语原文、读音标注（假名/罗马音等）
4. **可编辑、可自建**：AI 给出的每条候选答句用户都能直接改（改意思、改目标语、改读音）；也能**自己写新的回答选项**，不局限于 AI 生成的 N 条
5. **用户决策**：从（改过或自写的）候选里选一句并开口说（练语言，不是 AI 代说）
6. **可选**：识别用户念了哪句 / 是否念对，写回对话记忆

神奈子：晚上可补流程图；她认为这是**核心功能**。

### 候选答句的编辑与自建（交互补充）

AI 生成的是**起点**，不是定稿：

| 能力 | 说明 |
|------|------|
| **编辑 AI 候选** | 每条候选的三层内容（中文意思 / 目标语 / 读音）均可就地改 |
| **自建选项** | 用户可新增空白或从某条复制后改的「自己的回答」 |
| **为何重要** | 真人场景里 AI 语气、敬语、店规用语常差一点；让用户改到能开口念，比强行多生成几条更实用 |

默认仍由 AI 先出 N 条；用户改完或加完后再选一句念。

### 可衍生的两个「单人版」

| 衍生 | 说明 |
|------|------|
| **语音陪练** | 像普通 AI 语音对话，AI 也参与说话；相当于练习；AI 被赋予**角色前提** |
| **文字对话** | 同场景的文字版练习 |

→ 核心是「真人助攻」；单人版是练习场，不是主卖点。

### 刻意不做的事

- 不做「双方都开翻译、各说母语」的双向同传
- 不做 AI 直接替用户说话（主路径是用户开口）
- **不做手机 App / 不做通话监听**（已对齐 pass）
- **不做端侧塞满 ASR+LLM+TTS**（能力与设备都不现实；在线即可）

### 讨论中的目标场景

| 场景 | 诉求 | 结论 |
|------|------|------|
| **当面口语（便利店等）** | 听对方 → 出候选答句 | **MVP 最高优先级**（线下场景，不是「速度」本身） |
| 电话（iPhone） | 后台听通话 | **pass**：系统级做不到就砍 |
| Teams 等会议 | 监听程序/网页对方声音 | 有技术路径，**非 MVP**；好奇 Mac API |

神奈子补充：用户若「手里只有一个手机」做当面场景会别扭 → 倾向**桌面端**交付。

## 产品形态：几次收敛

### 1）值不值得独立 App？相对 ChatGPT 语音

- 拼「跟 AI 聊天的智商」→ **不值得**，ChatGPT 实时语音更强
- 拼「真人在场那几秒：听懂但不会说」→ 有独立切口
- 「多候选 + 中文/日语/读音、可点选、可编辑、可自建」= Chat UI / generate UI 能做的事，整体更像 **chat UI / 语言助手壳**，不是重型平台

### 2）AdventureX 交付形态（下午对齐）

| 决策 | 内容 |
|------|------|
| 手机 App | **不做** |
| 通话监听 | **pass** |
| 端侧离线全家桶 | **不可能**（ASR + LLM + TTS）→ **在线** |
| 网页 vs 打包 | 活动主旨要「独立项目感」→ **Mac App**；形态参考 Figma（见下），不是简单网页套壳 |
| Airi 插件（活动期间） | 路路曾提复用 Airi 能力；神奈子反感「挂在别人项目上交差」→ **活动中不做**（会后可插件化） |
| MVP | 路路：先解决一个 MVP；优先级 = **线下当面「听对方 → 多候选自念」** |

### 3）「像 Figma 那种」——她真正指什么（13:57 澄清）

**不是** Figma 插件，而是 **Figma 这种产品形态**：

- 纯在线平台，有网页版 + Mac 版
- Mac 程序**不是**简单网页套壳：本地放好基础 UI、按钮、窗口框架
- 具体显示什么文字/图片，**联网获取**
- 路路：只有明确需要原生能力时，才做原生部分；其余套壳是成熟方案

### 4）原生 vs 套壳 vs UI 风格（14:00–14:03）

| 话题 | 结论 |
|------|------|
| 「原生」 | 路路确认 ≈ 苹果官方语言栈（Swift 等） |
| 何时做原生 | **仅当有明确原生需求**（如 ScreenCaptureKit 抓 Teams 音频）；MVP 当面麦克风不必 |
| 系统组件 | 神奈子：苹果自带系统组件**用不上**；她要自己做 UI |
| 视觉方向 | 个性化、风格化；提到 **Liquid Glass** 一类质感（毛玻璃/流体玻璃风） |
| 分工暗示 | 她偏 UI/视觉；壳层用 Tauri/Electron + Web 自绘 UI 也能做 Liquid Glass，不必为 UI 全上 Swift |

**技术路线倾向**：Figma 式「轻原生壳 + 在线内容」或 Tauri/Electron 打包自研 Web UI；原生 Swift 模块按需加（会议音频等），不为好看 UI 硬上原生。

### 5）插件分歧：路路要复用，奈子要独立壳

**路路的真实动机（不是「为了做插件而做插件」）**

- Airi（或同类宿主）里很多能力已经有了：语音链路、对话壳、角色、模型接入等
- 做成插件 / 挂在已有栈上，是为了 **直接复用，少自己再搭一遍 ASR·对话 UI·会话管理**
- 核心增量仍是「真人听写 → 多候选可念答句」；底座能借则借

**神奈子的态度**

- 活动语境下很反感「挂在别人项目上交差」（独立项目感、指南理解）
- 看到「插件板块特别奖」截图后抗拒表情明显
- 可接受边界：**活动结束后**若对方想要这功能，再做成插件也行

**可调和做法**

- 活动交付：独立可演示的 Mac/Web 壳（满足她）
- 工程上：能力用 **现成 API / 开源模块 / 同一套业务逻辑**，不必从零造轮子（满足你）
- 「复用」≠ 必须挂 Airi；云端 STT/LLM、开源 chat 壳、自研一页 UI 都能复用
- 会后：同一核心逻辑再包一层 Airi 插件分发

## 调研摘要（技术）

### 1. 市面相近品

- AI 陪练 + 建议：HaruSpeak、IKI、XaiTok、Banter、Sayoo（对象多为 AI）
- 同传/通话翻译：偏听懂/代说
- 会议转写：偏纪要，少「可念候选」

### 2. Web vs 原生

核心逻辑 Web 可做；卡点是音频来源。

### 3. 按场景

| 场景 | 可行性 |
|------|--------|
| 当面麦克风 | **高** → MVP |
| iPhone 通话 | **极低** → 已 pass |
| Teams：Chrome 共享标签页音频 | 中高 |
| Teams：macOS ScreenCaptureKit | 高（需屏幕录制权限；可比 BlackHole 少装驱动） |
| BlackHole 虚拟声卡 | 可行但安装重（神奈子搜到的答案主推这条；现代方案可优先 ScreenCaptureKit） |

注：神奈子说「Mac 系统录屏不走声卡、是静音的」——指的是**普通屏幕录像不一定带系统声**；要用 ScreenCaptureKit 显式抓音频，或虚拟声卡绕路，不是「Mac 永远听不到 App 声音」。

### 4. 总判断

| 问题 | 答案 |
|------|------|
| 有没有价值？ | 有，切口是真人开口助攻 |
| 值不值得重 App？ | 交付打成 Mac 壳即可；内核是 chat UI + 在线 API |
| v1 做什么？ | **线下当面 / 桌面麦克风 → 多候选答句** |
